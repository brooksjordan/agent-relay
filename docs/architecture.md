# Ship While You Sleep — Architecture

## Three-Tier Model

```
C:\computronium\              THE BRAIN
  Methodology, philosophy,    Governs ALL projects and processes.
  model wrappers (GPT/Gemini) Not specific to build-while-you-sleep.
       |
C:\ship_asleep\               THE SUB-BRAIN (Tier 2)
  PowerShell scripts,         Governs the build-while-you-sleep process.
  templates, docs,            Called FROM here, never copied INTO projects.
  lessons learned             Invoked: launch-auto-compound.ps1 -ProjectPath "C:\<project>"
       |
  +----+----+
  |         |
C:\test_   C:\agent_          PROJECTS (Tier 3)
orchids\   roi\               Use the Brain + Sub-brain to achieve goals.
                              Contain: project code + build artifacts
                              Do NOT contain: build scripts
```

### What lives where

| Tier | Directory | Contains | Does NOT contain |
|------|-----------|----------|------------------|
| Brain | `C:\computronium\` | Methodology, CLAUDE.md, model wrappers, metrics | Project code, build scripts |
| Sub-brain | `C:\ship_asleep\` | Pipeline scripts, templates, docs, lessons learned | Project code, project-specific config |
| Project | `C:\<project>\` | Source code, priority reports, tasks, logs, PRDs | Pipeline scripts (those live in ship_asleep) |

### Invocation pattern

Ship_asleep scripts are always called **from** `C:\ship_asleep\scripts\`, pointing **at** a project path.

**MANDATORY: Use the launcher script.** It spawns a visible window and sets the required env var. auto-compound.ps1 will refuse to run without it.

```powershell
# CORRECT — the ONLY way to launch the build pipeline:
C:\ship_asleep\scripts\launch-auto-compound.ps1 -ProjectPath "C:\agent_roi" -Verbose

# Dry run (verify without building):
C:\ship_asleep\scripts\launch-auto-compound.ps1 -ProjectPath "C:\agent_roi" -Verbose -DryRun

# WRONG: running auto-compound.ps1 directly (will be rejected — missing env var)
# WRONG: copying scripts into the project
# WRONG: running scripts from inside the project directory
# WRONG: running hidden or as a background job
```

### What the pipeline creates inside a project

```
C:\<project>\
  reports/
    PRIORITIES.md     Active priority report (the ONLY file the pipeline reads)
    archive/          Completed/old priority reports (ignored by pipeline)
  tasks/              PRDs and task JSON files (created by auto-compound stages 4-5)
  logs/               Pipeline logs (created by auto-compound)
```

These artifacts are build outputs, not pipeline code. They belong to the project.

### Report Lifecycle

1. **Active report:** Always `reports/PRIORITIES.md` — this is the only file the pipeline reads
2. **Archive:** When starting a new phase, move the old report to `reports/archive/YYYY-MM-DD-<slug>.md`
3. **Commit before launch:** The report MUST be committed and pushed to origin/main before launching the pipeline. Stage 0 preflight verifies this via `git show` and will abort if the report isn't on the remote.

## Tier Contracts

### Priority Report Format

The pipeline reads exactly `<project>/reports/PRIORITIES.md` with this structure:

```markdown
## Priority 1 [ID]: Title
Description of what to build.
### Acceptance Criteria
- Specific, testable criteria

## Priority 2 [ID]: Title
...
```

Completed priorities should be struck through: `## ~~Priority 1 [ID]: Title~~ COMPLETE`

The `analyze-report.ps1` script reads this file and extracts the first non-completed priority.

### Task JSON Format

Generated by Stage 5 of auto-compound. Stored in `<project>/tasks/`:

```json
{
  "prd_source": "prd-feature-name.md",
  "created_at": "ISO-8601 timestamp",
  "tasks": [
    {
      "id": 1,
      "title": "Short task title",
      "description": "What to implement",
      "acceptance_criteria": "Specific, testable criteria",
      "status": "pending"
    }
  ]
}
```

### Pipeline Stages

| Stage | What it does | Destructive? |
|-------|-------------|--------------|
| 0 | Preflight: env var, clean tree, report on remote, open priorities | No (runs before reset) |
| 1 | Git reset to origin/main + clean prior build artifacts (tasks/, logs/) | YES - resets working tree |
| 2 | Load PRIORITIES.md, analyze for #1 priority | No |
| 3 | Create feature branch | No |
| 4 | Generate PRD from priority description | No |
| 5 | Convert PRD to task JSON | No |
| 6 | Execute tasks via loop.ps1 | Writes code |
| 7 | Push branch, create PR | No |
| 8 | Mark priority complete in report, commit + push to main | Modifies report on main |

**Stage 0 protects against accidental data loss.** It checks for uncommitted work and aborts before Stage 1 can destroy it. It also verifies the report exists on the remote, preventing the "uncommitted report wiped by reset" failure mode.

**Stage 1 is destructive.** It runs `git reset --hard` and `git clean -fd`, then removes gitignored build artifacts (`tasks/` and `logs/` directories) from the prior run. Stage 0 ensures this is safe before it runs.

**Stage 8** switches back to main after the PR is created and automatically marks the completed priority with strikethrough + completion note in the priority report. This eliminates the manual step of updating the report after each build.

## Versioning Policy

### Stable releases get tagged

When a version of the pipeline is confirmed working on a real project, tag it:

```powershell
git tag v2-stable  # The version that ran test_orchids successfully
```

### Experimental work goes on branches

Pipeline improvements are developed on feature branches (e.g., `v3-experimental`) and tested on a canary project before merging to main.

### Never ship experimental changes directly to main

The overnight pipeline runs from `main`. Untested changes to main = broken overnight runs.

### Test sequence for pipeline changes

1. Develop on a feature branch
2. Test on a small/test project (e.g., test_orchids)
3. Confirm full pipeline completes (all 9 stages: 0-8)
4. Merge to main
5. Tag the new stable version

## Native Command Error Handling

PowerShell's `$ErrorActionPreference = "Stop"` treats ANY stderr output as a terminating error — even when the command succeeds. Native tools like `git`, `gh`, and `claude` routinely write progress info to stderr.

**Use the `Invoke-Native` helper** (defined in auto-compound.ps1):

```powershell
$result = Invoke-Native git push -u origin $branch
if ($result.ExitCode -ne 0) {
    Write-Log "git push failed (exit $($result.ExitCode)): $($result.Output)" "ERROR"
}
```

This helper wraps `$ErrorActionPreference = "Continue"` + `$LASTEXITCODE` into a clean function. Use it for all native command calls within auto-compound.ps1.

## Known Risks

- **Mid-run upgrades**: If ship_asleep/main changes while a pipeline is running, the behavior is undefined. Pin to a tag for safety.
- **Non-idempotent stages**: Re-running can create duplicate branches or PRs. Stage 3 handles existing branches, but Stage 7 may create duplicate PRs.
- **Stage 1 destructiveness**: `git reset --hard` + `git clean -fd` on the project. Any uncommitted work is lost.

## Mandatory Requirements

- **Use the launcher**: Every build MUST use `launch-auto-compound.ps1`. It spawns a visible window and sets the env var that auto-compound requires.
- **Report on remote**: `reports/PRIORITIES.md` must be committed and pushed before launching. Stage 0 verifies this.
- **Clean working tree**: No uncommitted changes. Stage 0 verifies this.
- **Native command safety**: Use `Invoke-Native` for git/gh/claude calls. Never rely on stderr.
